```

  raw/          RH20T_cfg1 ŌĆ” RH20T_cfg7   untarred scenes; calib+joint patch merged into cfg1/2
```

**The joint.npy vector is NOT "N joint angles"** ŌĆö it concatenates position with
velocity (and torque on KUKA). Joint *positions* are always the first `DOF` values
(`robot_joint_field` in rh20t_api's `configs.json`). Anything that consumes joints must
slice by DOF, not take the whole vector.

Consistent across all cfgs (verified per scene):
- `tcp_base.npy`: per-ts dicts, `tcp` is always **(7,)** pos+quat; `robot_ft` (6,).
- `force_torque_base.npy`: per-ts dicts, `zeroed`/`raw` always **(6,)**. Present even
  for cfg5 (Franka, no external sensor) ŌĆö those values are robot-internal estimates,
  treat with caution.
- `gripper.npy`: `{serial: {ts: {gripper_command: [width, ŌĆ”], gripper_info: [ŌĆ”]}}}` ŌĆö
  `gripper_command` present everywhere. Gripper *hardware* differs (3 models), so raw
  widths are not directly comparable across cfgs.
- Camera count per scene: 7-10 (varies within a cfg, e.g. cfg4 has 7 and 8).
- cfg7 additionally has `transformed/tactile.npy` (unique to it).
- `audio_mixed/`: present in all cfgs but **not every scene** (some cfg1/2 scenes have
  0 wavs; some have 2+ segments).

## Anomalies / traps

- **`_human` vs `_human_2` naming**: human demos come in `ŌĆ”_human` AND `ŌĆ”_human_2`
  variants. `extract_frames._is_scene` filters with `endswith("_human")`, so the 543
  `_human_2` scenes **leaked into frames/ and the stage-1 shards** (frames dirs Ōēł robot
  + `_human_2`). Filter with `"_human" in name` instead. Impact: ~4% extra (robot-less)
  video in stage-1 pretraining ŌĆö harmless for video-only JEPA, but stage-2 state
  loading must exclude them (they have no robot state).
- **57 robot scenes (~0.4%) lack `transformed/joint.npy`** and must be skipped:
  cfg1: 3, cfg2: 4, cfg3: 1, cfg4: 17, cfg5: 4, cfg6: 8, cfg7: 20.
  Full list: `preprocessing/analyze_cfgs.py --full-scan`.
- A few scenes have cameras without `color.mp4` (extract_frames already skips them);
  frames dir counts are within a handful of the expected scene counts (cfg1 ŌłÆ1,
  cfg2 ŌłÆ1, cfg4 ŌłÆ6).
- cfg5 F/T is not from a physical sensor (see above).

## Implications for stage-2 (mm_perceiver) training beyond cfg3/4

The current state pipeline hardcodes cfg3's UR5 shape and breaks on 5 of 7 cfgs:

- `world_tokenizer/state.py`: `SceneState.state()` sin/cos's the **whole** joint vector
  ŌåÆ 28 dims only when it's 6 long; cfg1/2/5 give 44, cfg6/7 give 58. `STATE_DIM=28`
  and `FT_DIMS=range(21,27)` are absolute offsets that shift with any joint change.
- `world_tokenizer/step1_gate.py` `GROUPS` slices assume the same 28-dim layout.
- `world_tokenizer/precompute_patch.py` hardcodes `RAW=ŌĆ”/raw/RH20T_cfg3` independent of
  `--frames-root` (pointing frames at another cfg would silently pair its images with
  cfg3 state) and would cache whatever dim `state()` returns, unchecked.
- `world_tokenizer/train_perceiver.py` constructs `MMPerceiver()` without `state_dim`,
  so the model expects 28 regardless of the cache.

These are resolved by the chunked pipeline below (`chunk_state.py` ŌåÆ
`preprocessing/precompute_chunks.py` ŌåÆ `dataloader.py`); the legacy 28-dim path
(`state.py:SceneState`, `precompute_patch.py`) remains cfg3/4-only.

## Timing model: ticks, rates, and gaps (verified by probes)

- The four 10 Hz streams (`joint`, `gripper`, `force_torque_base`, `tcp_base`) are
  **camera-aligned**: per serial, their timestamps are *identical* to the camera color
  timestamps. The shipped data is NOT at native sensor rate (the paper's table
  describes the raw logs).
- The tick rate is nominally 10 Hz but really **6.7ŌĆō14.7 Hz, varying by cfg and by
  scene** (cfg3/4 Ōēł 7 Hz, cfg2/5 Ōēł 14 Hz, cfg6 spans 10.1ŌĆō14.5 Hz across scenes),
  with jitter within scenes.
- `high_freq_data.npy` (wrist F/T + TCP only, under the `"base"` key) is **100 Hz for
  cfg1 but 125 Hz for cfg2/3/4/6/7**. It is missing for essentially all of cfg5
  (file absent in 1298/1321 scenes, empty otherwise ŌĆö no physical F/T sensor) and
  empty for ~Ōģō of cfg3 scenes.
- Some scenes contain **multi-minute recording gaps** in their timestamps (e.g. 84 s
  in a cfg2 scene, 5 min total in a cfg5 scene) ŌĆö split into contiguous segments at
  gaps > 500 ms before chunking.

## Stage-2 chunk packet (chunk_state.py / precompute_chunks.py / dataloader.py)

Chunks are **tick-anchored** (chunk k = k-th tick of a contiguous segment; native
samples only, no interpolation). `dataloader.py` yields dict batches:

| key | shape | contents |
|-----|-------|----------|
| `rgb` | [B, 1, 196, 768] | frozen ViT-B/16 patch tokens of the chunk's frame |
| `motor` | [B, 1, 8, 3] | rows 0-6 joints, row 7 gripper width; C = [sin q, cos q, symlog dq]; gripper row ch0 = symlog width. No torque channel (KUKA-only ŌåÆ dropped) |
| `motor_mask` | [B, 8, 3] | UR5: row 6 + vel channel masked (23.4% of data); gripper row ch0 only |
| `ee` | [B, 13, 15] | high-freq samples in [tick k, tick k+1): [symlog F/T (6), symlog tcp xyz (3), tcp 6D rot (6)], zero-padded to T=13 |
| `ee_mask` | [B, 13] | all-False for cfg5 + hf-empty cfg3 scenes (~12.4% of data) |
| `robot_id` | [B] | 0=flexiv(cfg1/2) 1=ur5(cfg3/4) 2=franka(cfg5) 3=kuka(cfg6/7) |
| `cfg`, `scene_idx`, `group_idx` | [B] | lookups via `dataset.scenes` / `dataset.groups` |
| `ts` | [B] | tick timestamp (ms) ŌĆö for ╬öt-based triplet selection (metrics/METRICS.md) |

Caches: `caches/cfg<N>.npz` on the NAS, written by
`preprocessing/precompute_chunks.py` (`--chunks-per-scene`, resumable per cfg);
loaded/concatenated by `world_tokenizer.dataloader.make_loader`.

**Train/test split: held out by (cfg, task, user) group, stratified per cfg.** The
same user repeats the same task ~10├Ś (`scene_0001ŌĆ”0010` ŌĆö median exactly 10
scenes/group, 12,776 scenes ŌåÆ 1,293 groups); those recordings are near-duplicates, so
splitting per scene would leak ~9 training near-copies of every test scene. All of a
group's scenes land on one side; each cfg holds out ~30% of its groups (so tier-4
"different config" triplet negatives exist for every robot). For the metrics suite
(`metrics/METRICS.md`): compute on the test loader only, and draw triplet negatives
from test groups only ŌĆö train-scene negatives would mix encoder-seen material into
the eval.
